# Factory-Shell Migration — Master Plan (handles all risks; completes the move to the factory approach)

This is the consolidated, in-repo execution plan to finish the Factory pattern: delete all 45 per-step `<Name>StepBuilder` shell classes and let one shared `TemplateStepBuilder` facade + the `PatternHandler`/`strategy_registry` factory drive every step from `.step.yaml` data. It SUPERSEDES `2026-06-29_design_b_classless_rollout_plan.md` (which scoped only the step-catalog provider seam + a narrow validation chokepoint and prematurely marked the work near-done). It is grounded in two adversarially-verified analyses in the abuse_slipbox FZ trail: **FZ 31e1d3g3** (the 6-system, 32-confirmed-risk cross-system register) and **FZ 31e1d3h** (the alignment↔validation redefinition). Every file:line below was verified against live source on 2026-06-29.

## Why this plan exists (what the old plan got wrong)

The old plan marked Phases 0/1/2 "done," but the g3 audit found Phase 2 shipped only HALF of itself: the lazy `_BuilderDescriptor` (commit `8fec1a98`) decoupled the AST scan from importlib I/O, but the actual classless mechanism — a **registry-walk materializer that synthesizes a builder for a step with NO physical file** — was never built. Discovery still globs `builder_*.py` (`builder_discovery.py:351`) and `steps/builders/__init__.py:15-160` still eager-imports all 45 names. So today, deleting the 45 files reds discovery + package-import + validation simultaneously. The old plan also under-scoped validation: it assumed a single "materialize the class once" chokepoint suffices, but g3+h proved the validators ALREADY misfire on the shells that exist today (the `getsource`/`hasattr` checks are semantically dead against a shared shell), so the validation work is a redesign (3 boundaries), not a chokepoint patch.

The reframe that makes this tractable: **the 45 builders are ALREADY pure declarative shells** — each is ~16 LOC, `class XxxStepBuilder(TemplateStepBuilder): STEP_NAME = "Xxx"`, and ZERO define `create_step`/`_get_inputs`/`_get_outputs`/`_create_*` (42 are methodless; only dummy_training, cradle (2 each), bedrock (1) retain residual Python). The factory is realized BEHAVIORALLY; what remains is deleting 45 NAMED classes that still serve only as discovery handles + instantiation targets, while keeping every gate green. The seam is sound (proven): `BuilderProvider = Callable[..., Any]` (`step_catalog/models.py:22`), the assembler call `builder = builder_cls(**5kwargs)` at `pipeline_assembler.py:189` has NO isinstance/issubclass gate (re-verified), and the FZ 31e2 triangle-closure gate ships. The work is to build the deletion MECHANISM and re-base validation on data — in an order that never reds a gate.

## Phase A — Materializer + residue (MUST precede any deletion)

This is the load-bearing phase; every "high" risk depends on it.

- **A1 — registry-walk materializer (closes #1/#2/#10/#11).** Add a registry-walk provider source to `BuilderAutoDiscovery`: for any `STEP_NAME` carrying a `.step.yaml` interface and NO physical file, synthesize a real per-step subclass `type(f"{step}StepBuilder", (TemplateStepBuilder,), {"STEP_NAME": step})` and cache it (`lru_cache`-style, per step_name, so identity is stable within a process — see OQ 31e1d3g3a). The synthesized SUBCLASS (not `functools.partial`, not a bare lambda) is required because `TemplateStepBuilder.__init__` reads `self.STEP_NAME` to call `load_step_interface` (`builder_templates.py:986-991`) and `_auto_bind_handler` (`:1003`), and because a real subclass preserves `.__name__`, satisfies `issubclass`/`__mro__`, and survives the #4 step-name collapse for free. `load_builder_class` tries file-descriptors FIRST then the synthesized provider (`builder_discovery.py:223-263`), so deletion is incremental and any surviving file wins. Drive the walk from the interface-having + routable subset (steps whose `sagemaker_step_type` has a registered `PatternHandler`), EXCLUDING `_INTERFACE_LESS = {Base, Processing, HyperparameterPrep}` and the `register_no_builder` rows (`builder_templates.py:915-916`), so abstract rows don't fabricate providers and trip the orphan/bijection tests. Keep the 4 SDK steps undiscoverable OFFLINE (defer the SDK import to instantiation) so `get_builder_map` still omits them offline. Fold the synthesis into `discover_builder_classes` and the job-type-variant fallback (`mapping.py:111-120`).
- **A2 — SDK knob residue (closes #18).** `sdk_step_class` is a live SAIS `*Step` class reference that `PatternsSection.as_knobs()` (`step_interface.py:336-345`) cannot serialize and `SDKDelegationHandler.build_step` requires (`builder_templates.py:858-863`). Add a small `sdk_bindings.py` mapping `STEP_NAME -> lazy thunk` (each value does a LOCAL import, never module-level — that is exactly why those steps are offline-undiscoverable), consulted by the materializer to inject the knob via the existing `knobs.update()` merge (`builder_templates.py:1028-1030`). Derive the carve-out set from the YAML (`registry.requires: secure_ai_sandbox_workflow_python_sdk` and/or `patterns.step_assembly == 'delegation'`), not a second hardcoded list; add a CI assertion `{steps with step_assembly==delegation} == set(sdk_bindings.keys())`. Keep the `NotImplementedError` guard as the honest failsafe.
- **A3 — re-home the 3 residual builders' Python.** dummy_training's 3-tier input resolution → a handler input-mode knob; bedrock_batch's `_get_environment_variables` runtime-S3-path computation → an env-from-base-path handler knob; cradle's `get_output_location`/`get_step_outputs` are PUBLIC post-build introspection helpers called by external exe_doc/assembler code — treat as their own external-API migration (relocate onto the built-step object + update callers, OR keep a thin retained cradle shell; see OQ 31e1d3g3c).
- **Exit gate:** the registry-walk materializer yields a provider for EVERY step_name discovery currently produces (set-equality before/after, asserted in a test); closure gate + resolved-edge-graph snapshot green; no file deleted yet.

## Phase B — Import surface (closes #3)

`steps/builders/__init__.py:15-160` eager-imports the 45 concrete names into `__all__`; deletion → ImportError at package import + breaks the public API surface. Fix at BOTH coupling points with ONE synthesis source: replace the ~37 unconditional `from .builder_x import X` (`:15-114`) and the hand-maintained `__all__` (`:117-160`) with a PEP-562 module `__getattr__` (precedent at `core/base/__init__.py:45-52`) that lazily synthesizes the SAME `type(name, (TemplateStepBuilder,), {"STEP_NAME": …})` and derives `__all__` from the registry. A module `__getattr__` alone does nothing for the filesystem-reading discovery layer — that is A1's job; this is the import-surface twin. Migrate the 3 module-path test imports (`tests/core/base/test_builder_migration_parity.py:97`, `test_compute_spec.py:78`, `test_training_handler.py:166`) off the deleted modules. **Exit gate:** `__init__` names ZERO builder modules statically; `import cursus.steps` succeeds with the 45 files still present.

## Phase C — Core step-name + facade + registry hygiene

- **C1 — step-name collapse (closes #4, the hardest runtime blocker).** `builder_base._get_step_name()` (`:361-371`) derives the canonical name from `self.__class__.__name__[:-11]`; without the named shells it collapses to "Template" → colliding step/job names. Lift the preference: `canonical_name = getattr(self, 'STEP_NAME', None)` falling back to the `__class__.__name__` strip only when `STEP_NAME` is falsy — one edit transparently fixes `_generate_job_name` (`:406`) and the 5 handler call sites (`builder_templates.py:346,511,593,752,871`). Add a hard guard in the facade that RAISES if `STEP_NAME` is None at build time for a routed shell (converts a silent "Template" collision into a loud failure), plus a parity test exercised THROUGH the provider path (`get_builder_for_config` → instance), not just by instantiating a subclass.
- **C2 — facade-routing + construction-aware gate (closes #13/#19).** Extend the 31e2 gate so that for every concrete ROUTABLE registry step it asserts both (a) `load_step_interface(step)` loads (covering each declared job_type variant) and (b) `resolve_handler(get_sagemaker_step_type(step), step_assembly)` yields a routable handler — deriving routability + `step_assembly` from the registry/`.step.yaml` patterns, NEVER a hardcoded verb list. Decide (OQ 31e1d3g3b) whether the gate asserts handler-resolves + config-coverage STATICALLY or actually instantiates the provider through `get_builder_for_config` and calls `create_step` — prefer static to avoid re-importing the `_create_minimal_config` fragility into the gate.
- **C3 — registry hygiene (closes #12/#24/#25).** Collapse the two stale hand-maintained SageMaker-type sets (`validation_utils.VALID_SAGEMAKER_TYPES` + `step_names.validate_sagemaker_step_type`) into ONE constant = union of live `get_all_sagemaker_step_types()` + a small static floor (`Base, Utility, RegisterModel, Lambda`); add `config_class: BatchTransformStepConfig` to `batch_transform.step.yaml` and delete `_CONFIG_CLASS_OVERRIDES` in the SAME atomic commit (golden snapshot diff reviews as a no-op); delete the dead `STEP_ASSEMBLY` class-attr fallback (`builder_templates.py:966,1022`) and add a conformance assertion that every Processing-type `.step.yaml` declares `patterns.step_assembly` explicitly.
- **Exit gate:** step-name parity test green through the provider path; construction-aware closure gate green; registry golden snapshot unchanged.

## Risk 2 (workspace-discovery regression) — the deliberate decision

DECISION: a PERMANENT HYBRID with an opt-in flag — registry-walk for the package, retained AST filename-fallback for workspace dirs, gated behind `allow_unregistered_workspace_builders` (default True for workspace, False for package), with the `.step.yaml` stub as the PROMOTE path, never a run precondition. Rationale: requiring a stub up-front is the velocity-killer (a dev dropping `builders/builder_my_experiment_step.py` would have to author registry/interface YAML before iterating, defeating the convention-pickup at `builder_discovery.py:489,509`); a pure flag with no retention under-serves the common workspace case; the hybrid keeps the package strictly registry-gated (the SSOT the migration wants) while leaving the workspace escape hatch on — structurally clean because package and workspace already use separate methods (`_discover_package_builders:289` vs `_discover_workspace_builders:298`). Implementation: gate the unregistered-name fallback RETURN (`builder_discovery.py:489/509` — return None when the flag is off and no registry match), not the directory scan; emit a loud WARNING when a convention name is returned; consult workspace descriptors BEFORE any registry-closure gate (`load_builder_class:244-252` already checks workspace first); scope the triangle-gate's "discovered ⊆ registry" assertion to PACKAGE builders only. Note: discovery keys off the `<Name>StepBuilder` class-name suffix (`_get_step_name` at `builder_base.py:361`), not `STEP_NAME`, so a workspace builder needs only that name + a `StepBuilderBase`/`TemplateStepBuilder` base.

## Sequencing rationale (why this order, not another)

The risks couple through three file-dependent surfaces — discovery, `__init__` exports, validation — so deleting the 45 files before the replacements ship reds all three at once (#20). A enables everything (nothing can be deleted until a fileless step still produces a builder). B makes `import cursus.steps` survive deletion. C makes the synthesized builder name + route correctly (else every step collides on "Template"). D re-bases validation on data (else the gate flips to hard-fail on the first deleted file). Only with A–D green is E (the deletion) safe, and even then batched + byte-diff-gated so any single batch is reversible. The whole sequence keeps the assembler/build flow untouched (the `builder_cls(**5kwargs)` call at `pipeline_assembler.py:189` is duck-typed and invisible to class-vs-provider), so a partially-migrated tree always compiles and builds.
