# Phase D — Validation 3-Boundary Redesign: Verified Implementation Change-List

This is the file:line-precise, adversarially-verified implementation spec for Phase D of the factory-shell migration (master plan `2026-06-29_factory_shell_migration_master_plan.md`, design FZ 31e1d3h). It was produced by a 53-agent mapping workflow that traced six change surfaces, proposed 46 concrete edits, and adversarially verified each against live source (36 safe, 10 required ordering/mechanism corrections that are folded in below). All file:line anchors were re-verified against live source on 2026-06-29. This note exists to capture the finding while offline; implementation follows it gate-by-gate (D0→D5), each a small commit that keeps the FZ 31e2 closure gate, the alignment CLI suite, and the validator unit suites green.

## Why Phase D is repairing live breakage (empirically confirmed)

Two misfires were reproduced against live code, not just predicted by the design. First, the L4 step-type validators report working shells as FAILED: `TrainingStepBuilderValidator.validate_builder_config_alignment("XGBoostTraining")` returns `status=FAILED` with `"Missing Training required method: _create_estimator"`, because the validator does `hasattr(builder_class, "_create_estimator")` which is FALSE on the declarative shell — the estimator factory lives in `TrainingHandler.make_compute` (`builder_templates.py:524-537`), not the shell. grep confirms ZERO shells define any `_create_*` method, so EVERY shell is a false-negative today. Second, L2 reports `"No specification files found for XGBoostTraining"` via `step_catalog.find_specs_by_contract` (stale spec discovery from the deleted per-step `*_spec.py` world). So the 4-level validators are already broken for the shell architecture; Phase D is the repair, and the master-plan exit gate (`master_plan.md:82`) requires the 3-boundary suite + teeth-test green BEFORE any file deletion (Phase E), because the moment a `builder_*.py` is deleted the same path flips from spurious-noise to a hard import/attribute failure.

## The three governing facts the spec is built on

Fact 1 (ordering): the four `*_step_validator.py` modules + `validator_factory.py` are NOT dead today — they are reached at runtime via `unified_alignment_tester.py:217-221` → `level_validators.py:160` `run_level_4_validation` → `:241` `_get_step_type_validator` → `validator_factory.py:11-15` imports + `:50-58` registry, consumed by `cli/alignment_cli.py` and `mcp/tools/validate.py`. So the new B3 validator must be BUILT and wired FIRST (D1), and only then may the old validators + their pinning tests be deleted (D3) — a swap-to-nonexistent-B3 would fail the alignment subsystem at import. B3 is a NEW module (`registry_binding_validator.py` does not exist in src today), wired through the existing `run_level_4_validation(step_name, validator_class)` signature so MCP/CLI/`universal_test.py:268` are byte-unchanged.

Fact 2 (B3 mechanism): the validators receive the builder CLASS via `load_builder_class`, but `compute.kind` / `content_type` / `input_source_overrides` / the `*_field` descriptors live on the per-instance StepInterface loaded in `TemplateStepBuilder.__init__` via `load_step_interface(step_name, job_type)` — NOT on the class. So B3 must itself call `load_step_interface(step_name)`. The "can a step be ASSEMBLED" predicate that replaces the `hasattr` gate mirrors each handler's own disjunction (`builder_templates.py:317-325` Processing, `:526-536` Training, `:604-614` ModelCreation, `:740-750` Transform): pass iff `_overrides(b,'_create_X') OR '_create_X' in vars(b) OR (contract.compute and contract.compute.kind) OR resolve_handler(...) supplies make_compute`. Because no shell defines `_create_*`, every shell passes solely via the `contract.compute.kind` branch — verified present: XGBoostTraining/PyTorchTraining `estimator`, TabularPreprocessing `sklearn`, XGBoostModel `model`, BatchTransform `transformer`.

Fact 3 (the central B3 hazard the teeth-test must close): a descriptor-only coverage check FALSE-GREENS on the five Transform reads at `builder_templates.py:691-695` (`b.config.content_type/split_type/join_source/input_filter/output_filter`) — grep over all `.step.yaml` returns ZERO hits for `content_type`/`split_type`, yet they ARE read at build time and DO exist on `BatchTransformStepConfig` (`config_batch_transform_step.py:32,39,45,49,59`). So the coverage set must be the UNION of (a) descriptor-declared attrs (`compute.framework_version_field`, `compute.py_version_field` — both `Optional[str]`, so None values MUST be filtered out or they false-ERROR on Transform; `contract.input_source_overrides.values()`; `contract.job_arguments[].source` as a soft/WARNING signal since the real job-args teeth is the `config.get_job_arguments()` callable) AND (b) handler-declared `requires_config_fields` DATA (OQ 31e1d3h1's preferred route over an AST scrape, because `:220` `getattr(b.config, attr)` with a runtime `attr` and `:813` `getattr(b.config, fallback_attr, None)` with a knob value are statically undecidable). Config CLASS resolution must use `get_config_class_name(step_name)` (`step_names.py:254`), NOT `<Name>Config` mangling, because `batch_transform.step.yaml:5` breaks the convention. Missing a required attr = ERROR; missing a soft `.source` attr = WARNING.

Fact 4 (enum collapse safety): collapsing `ValidationLevel` is safe for runtime consumers (the scorer `scoring.py:13-17` keys on `alignment_validation`/`integration_testing`/`step_creation`, not level numbers, and iterates `validation_results.items()` dynamically; report keys are `level_{N}` at `unified_alignment_tester.py:271`) — but the Enum-alias trick is IMPOSSIBLE (a real alias shares a value, so aliasing `CONTRACT_SPEC` to 3 makes `ValidationLevel(2)` raise `ValueError`, breaking `validation_models.py:271`). Keep integer values `{1,3,4}` NON-contiguous and stable so `ValidationLevel(1/3/4)` coercion survives; `ValidationLevel(2)` is intentionally gone, and every `CONTRACT_SPEC` reference (source AND ~4 test files) must be removed in the SAME commit as the enum edit. The universal-Level-3 guard at `validation_ruleset.py:260-265` references `SPEC_DEPENDENCY` (=3, value kept) and survives unchanged.

## Refuted / guardrail items (the verifier's corrections, already folded in)

The 10 proposals flagged unsafe were corrected on ordering/mechanism grounds, not abandoned: (a) B3 must be BUILT as a new module and wired BEFORE the old validators are deleted — never an in-place swap of `_get_builder_class`'s return contract (that would break the method-presence callers mid-edit); (b) the enum collapse cannot use Enum value-aliases (a shared value breaks `ValidationLevel(2)`); it must drop CONTRACT_SPEC and edit every reference (source + the ~4 pinning test files) in ONE commit; (c) the per-type-validator test files are DELETED (they pin the removed false-negative on synthetic mocks), but the `test_universal_test.py` and `test_unified_alignment_tester_refactored.py` cases are REWRITTEN, not deleted (they are the surviving contract); (d) the importlib-fallback removal must be justified by "load_builder_class already wins," NOT "the file is deleted" (false until Phase E).

## What must NOT ship in Phase D (defer)

The actual deletion of the 45 shell files is Phase E (#61). Phase A3 (#58, re-home dummy_training/bedrock, retain cradle as a thin shell) is independent. Do not block D on either, and do not let any D edit assume a `builder_*.py` is already gone.
